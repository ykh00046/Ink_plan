# ink_plan 발전·보완 로드맵

> 작성: 2026-07-07 · 전체 코드 리뷰 + 엑셀/파생값 검증 직후
> 대상: 단일 Windows PC(LAN)에서 소규모 콘택트렌즈 공장 팀이 매일 쓰는 생산계획 도구
> 성격: **가동 중인 운영 도구** — "새 기능"보다 "안 멈추고, 손이 덜 가고, 데이터를 안 잃는 것"이 우선순위의 뿌리

## 1. 현재 상태 진단

### 강점 (지켜야 할 것)
- **순수함수 코어**(`data-service.js`)에 파생 로직 단일화 → 대시보드·페이지 배지·벨·사이드바가 항상 같은 수치. 테스트 199(Node)+Python로 보호됨.
- **백엔드 하드닝 완료**: 원자적 쓰기(temp+rename), OCC(If-Match, 이제 필수), 정적 경로 차단(translate_path 기준), DNS-rebinding/CSRF 방어, 64MB 본문 상한.
- **제품 정체성 id** 체계가 읽기·쓰기·집계 전 경로에서 일관(동명 8종 구분). 이번 리뷰로 잔여 결함까지 정리됨.
- **가져오기 2경로**: 엑셀 재고 조사표(SheetJS) + INK요청서 OCR(Gemini). 파싱→분류→가용일/부족까지 손계산과 일치함을 검증(2026-07-07).
- **변경 이력(audit)** + 3-way 동시편집 병합 → 다중 탭에서도 lost-update 방지.

### 구조적 제약 (발전의 병목)
| 제약 | 현상 | 영향 |
|---|---|---|
| **단일 스냅샷 저장** | `current.json` 2.1MB 통째 read/write, OCC도 전체 스냅샷 | 과거 조회(History) 보류, 추세/예측 불가, 저장 비용이 데이터에 비례 |
| **시계열 부재** | `inventory.daily`는 ISO 날짜 무한 누적, 주(週) 정렬은 M/D만 비교 | 연도 모호성(이번에 완화했으나 근본 미해결), 누적 성장 방치 |
| **빌드 없음** | 브라우저에서 ~13k줄 JSX를 매 로드 Babel 트랜스파일 | 로드 지연·미압축, 규모 증가 시 스케일 벽 |
| **단일 PC** | 백업이 전부 로컬(`data/backups/`, 매일 18:30) | PC 고장 = 단일 실패점, 오프사이트 사본 없음 |
| **관측 부재** | `log_message` 무음, 에러는 버려지는 stdout(VBS hidden) | 운영자가 실패를 못 봄, 사후 진단 불가 |
| **잉크 마스터 비대** | `inkPlan` 1598종인데 실수요는 ~114종 | 미사용 잉크 누적, lint/선택 UX 잡음 |

## 2. 우선순위 로드맵

원칙: **가동 신뢰성 → 매일 손 덜기 → 보류 기능 해금 → 분석 가치**. 각 항목에 영향(◎큼/○중/△작음)·노력(S/M/L) 표기.

### Phase 0 — 즉시 안정화 (1주 내, 저비용)
운영 리스크를 값싸게 낮추는 것들. 대부분 반나절 단위.

- **[◎/S] 오프사이트 백업**: 매일 백업을 네트워크 드라이브/클라우드 폴더로 복사(로보카피/rclone 1줄). *단일 PC 고장 = 전 데이터 소실*이 지금 가장 큰 실질 리스크.
- **[○/S] 저장 실패 관측**: 서버 에러를 `data/logs/server.log`로 회전 기록(현재 무음). 프론트는 이미 저장 실패 토스트가 있으나, 428/502 신규 응답 코드에 대한 사용자 안내 문구 점검.
- **[○/S] 미매핑 잉크 정리**: 현재 5월 조사표에서 `SONDER`·`FREEDOM`이 마스터에 없어 제외됨 → 등록 또는 오타 교정. (이전 FARAISE에서 바뀜)
- **[○/S] `inventory.daily` 아카이브 기준**: N주 지난 일자 값을 별도 아카이브로 분리(성장 억제 + 주 정렬 모호성 원천 축소).
- **[△/S] 남은 리뷰 low 항목**: 리뷰에서 남긴 낮은-우선 항목(메모리 `project-full-review-2026-07`) 소진.

### Phase 1 — 엔지니어링 기반 (2~4주)
회귀를 자동으로 막고 릴리스를 가볍게.

- **[◎/M] CI 도입**: GitHub Actions로 push 시 `node --test` + `python -m unittest` 게이트. 원격(origin)이 이미 있으니 즉시 가능. *하드닝·id 회귀를 사람 손 없이 차단*.
- **[○/M] 렌더 스모크 정식화**: 이번에 쓴 격리 서버 harness(임시 데이터 복사 + Playwright)를 `tests/`에 스크립트로 고정 → 대시보드·제품편집·사출셀·엑셀import 스모크를 재현 가능하게.
- **[○/M] 릴리스 프리컴파일(선택)**: 개발은 브라우저 Babel 유지, **릴리스만** `build_release.py`에서 JSX 사전 트랜스파일+병합 → 로드 지연·Babel 경고 제거. 개발 편의는 그대로.
- **[△/S] data-quality 확장**: 미사용 잉크(수요 0), 호기 미배정 잉크, 동명 그룹을 점검 리포트에 노출.

### Phase 2 — 데이터 모델 진화 (핵심 레버, 4~8주)
**여러 보류 기능의 뿌리.** 여기를 풀면 History·추세·예측이 연쇄로 열린다.

- **[◎/L] 주간 스냅샷 아카이브**: 매주 마감 시 그 주 계획/재고를 불변(immutable) 스냅샷으로 적재(`data/archive/2026-W28.json`). 
  - → **History view 부활**(현재 데이터모델 한계로 보류 중). 과거 주를 통째로 재현.
  - → 소비 추세·수요 이력 데이터 확보(Phase 4의 입력).
  - 감사 로그 replay보다 싸고 정확. `compareHistoryRows`가 이미 있으니 diff UI 재활용.
- **[○/M] inventory 시계열 정규화**: 주 정렬을 M/D가 아닌 ISO 주차 기준으로 → 연도 모호성 근본 해소(이번엔 "최신 ISO 우선"으로 완화만 함).
- **[○/M] 마스터 생명주기**: 잉크·제품에 활성/비활성 상태 → 비대한 1598 잉크에서 현역만 기본 노출.

### Phase 3 — 매일 손 덜기: 가져오기 정확도 (3~6주)
일일 워크플로의 실병목(검수 부담)을 줄인다.

- **[◎/M] OCR 정확도 루프**: 검수에서 사용자가 교정한 표기를 grounding 어휘로 피드백 → 같은 호기 반복 제품의 철자 수렴 강화. confidence 낮은 행만 검수로 라우팅.
- **[○/M] 사출 엑셀 직접 import**: 현재 사출은 OCR/수동만(엑셀 풀네임↔마스터 매칭률 4/43). `Sheet1.xlsx`(BOM) 판매명↔사출명 매핑을 매칭 레이어로 써 직접 import 가능성 검토.
- **[△/S] 약품 발주 추적**: 현재 인쇄 전용 → 발주 상태(요청/입고) 기록 옵션.

### Phase 4 — 분석·예측 (가치 확장, 데이터 축적 후)
Phase 2 스냅샷이 몇 주 쌓인 뒤 착수.

- **[○/L] 잉크 소비 추세 + 재발주점**: 주별 소비량 추세 → 재발주점(reorder point)·안전재고 제안. "곧 부족" 사전 경보를 사후(가용일)에서 예측으로 전환.
- **[△/L] 사출 이력 기반 수요 예측**: 과거 injection에서 잉크별 요일 수요 패턴 → 다음 주 소요 예측.
- **[△/M] 대시보드 확장**: 라인업+재고 위험 외에 주간 추세 카드.

## 3. 교차 관심사 (상시)
- **보안 하드닝 회귀 방지**: 정적 차단·OCC·호스트 가드는 테스트로 고정됨 → CI가 지키게.
- **성능 관측**: current.json이 커지면 저장 지연 체감 → 프리컴파일(Phase 1) + 스냅샷 분리(Phase 2)가 완화책.
- **문서**: `docs/RUNBOOK.md` 최신 유지(포트 8766, 백업/복원, 트레이앱). 신규 운영 항목(오프사이트 백업·로그 위치) 반영.
